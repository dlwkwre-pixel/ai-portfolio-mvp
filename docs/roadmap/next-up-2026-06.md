## 1. Site speed / navigation latency 🟠 IN PROGRESS
The #1 high-leverage item before onboarding strangers.

### Shipped
- **Lazy-loaded `xlsx-js-style`** (5ac6feb): was statically imported into planning-client but only
  used on the Excel-export click. Dynamic-imported inside `exportForecastXLSX` → /planning's
  largest chunk dropped **1.34MB → 862KB** (xlsx now a separate on-demand ~474KB chunk).
- **Parallelized per-portfolio valuations** (b24a516) on dashboard, planning, tax, portfolios:
  were serial `for await getPortfolioValuation` loops (render waited on the SUM of every
  portfolio's market-data latency) → `Promise.all` bounds it to the slowest single portfolio.
  Dashboard also parallelizes its per-portfolio holdings fetch.

### Remaining speed opportunities (next passes)
- **recharts (~8MB on disk)** still ships in the 862KB /planning chunk. Extract the forecast /
  net-worth chart blocks into child components and `next/dynamic(..., { ssr:false })` them so
  recharts loads only when charts are on screen. (Bigger refactor inside the 11k-line file — do
  carefully; likely the single biggest remaining bundle win.)
- **One holdings query instead of N**: dashboard still fires one holdings query per portfolio
  (now parallel, but still N round-trips). Switch to a single `.in("portfolio_id", ids)` + group
  in memory (planning/page.tsx already does this).
- **Prefetch audit**: landing/auth use `<a>` (no prefetch). In-app uses `<Link>` (ok). loading.tsx
  skeletons already exist so nav feels instant.
- **Shared quote fetch**: multiple portfolios re-fetch the same ticker quotes; a per-request quote
  cache across valuations would cut redundant Finnhub calls.
