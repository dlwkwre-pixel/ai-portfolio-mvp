### ⚠️ KEY CORRECTION — the "862KB chunk" is NOT the initial bundle

on-demand chunk that only downloads when a user clicks Export. It is NOT in the initial /planning

  chart route) + planning's own (smaller) app chunk. Extracting panels trims the app chunk ~25KB
  each — real but marginal, invisible behind the parked xlsx in the size ranking.
- CONCLUSION: per-panel extraction is now low-ROI (≈25KB each, refactor risk on the flagship).
  High-impact speed work is done (parallel fetches, single query, xlsx deferral, skeletons,
  prefetch). Further panel extraction = optional polish, not a real bottleneck.

Measured: the 862KB /planning chunk is `planning-client.tsx`'s OWN code (a ~5000-line main
component + ~21 top-level sub-component functions). Dynamic-importing the two already-separate
files (MasterLifeRoadmap, Plan401kSection) did NOT move the 862KB — they were already separate
chunks. To shrink it, each inline sub-component (EstatePlanningTab ~660 lines, OnboardingWizard,
CompareTab, BudgetTrackerTab, CashFlowOS, BalanceSheetOS, AiImportPanel, StatementImportPanel…)
must be MOVED to its own file + `dynamic()`-imported. Findings:
- The sub-components are top-level functions with clean props (no parent closures) + minimal
  shared deps (mostly a trivial `fmt` money formatter) → mechanically extractable, tsc-validated.
- BUT each is a 300-660 line block-move out of the flagship file; payoff ~30-50KB each, so it
  takes 6-8 extractions to meaningfully shrink → a multi-step refactor that should be done one
  component per commit WITH device testing between steps. High effort, modest per-step payoff,
  real regression risk if rushed.
RECOMMENDATION: do this as its own focused session, extracting one panel at a time (start with
